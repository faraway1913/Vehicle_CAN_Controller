# 车载CAN总线双机通信联调完整记录

**项目**：车载环境感知与数据采集系统 — STM32F407 + STM32F103 双机CAN通信
**日期**：2026-03-10 ~ 2026-04-06，历时约1个月
**状态**：✅ 全部完成

---

## 一、项目目标

实现 STM32F407（Master/ECU）与 STM32F103（Slave/仪表）的 CAN 总线双机通信：

| 项目 | 说明 |
|------|------|
| F407 | 采集油门电位器数据（ADC，200ms周期）和车内温湿度（AHT20，2s周期），封装为 CAN 标准帧发送 |
| F103 | 接收 CAN 报文，通过串口实时打印数据 |
| CAN 波特率 | 250 kbps（经过多轮调试最终确定） |
| CAN ID | 0x100（油门），0x101（温湿度） |
| 数据正确性 | ADC 值约 2400~2650（对应电位器旋钮），温湿度与传感器原始数据完全一致 |

## 二、硬件配置与接线

### 2.1 主控板

| 板子 | 型号 | 主频 | CAN 引脚 | 供电方式 |
|------|------|------|----------|----------|
| Master | STM32F407ZGT6（LQFP144） | 168 MHz | PA12(TX) / PA11(RX) | USB Type-C |
| Slave | STM32F103C8T6（LQFP48） | 72 MHz | PB9(TX) / PB8(RX)（重映射） | Micro - USB |

### 2.2 外设连接

**F407 侧（Master）**：
- AHT20 温湿度传感器 → I2C1（PB6 SCL / PB7 SDA）
- 电位器（油门模拟） → ADC1_IN1（PA1）
- TJA1050 CAN 收发器 → PA12(TX) / PA11(RX)，VCC 接 3.3V

**F103 侧（Slave）**：
- TJA1050 CAN 收发器 → PB9(TX) / PB8(RX)，VCC 接 3.3V
- 串口打印 → USART1（PA9 TX / PA10 RX），通过 CH340 连接电脑

### 2.3 CAN 总线接线规范

```
[F407 + TJA1050]  ←──── CAN_H / CAN_L ────→  [F103 + TJA1050]
       ↓                                              ↓
   120Ω 终端电阻                              120Ω 终端电阻
       ↓                                              ↓
      GND ─────────────────────────────────────── GND（共地）
```

**关键说明**：
- 总线两侧各并联一颗 120Ω 终端电阻，总线阻抗为 60Ω
- GND 必须直接互连（杜邦线直连，不能只靠电源间接共地）
- TJA1050 模块 VCC 接 3.3V（模块自带稳压电路，支持 3.3V~5V 输入）
- CAN_H / CAN_L 不能接反，两端一一对应连接

## 三、工具链说明

### 3.1 VSCode + CubeMX + Keil 协作开发流程

本项目使用三个工具协作开发，分工明确：

| 工具 | 作用 | 使用时机 |
|------|------|---------|
| **STM32CubeMX** | 可视化外设配置，生成初始化代码框架 | 初始配置、外设参数修改 |
| **VSCode** | 代码编辑、文件浏览、全局搜索 | 日常编码、查看代码 |
| **Keil MDK** | 编译、烧录、在线调试 | 编译固件、烧录到芯片、调试 |

**标准工作流程**：
1. 用 CubeMX 修改 `.ioc` 文件（配置外设参数、引脚分配、时钟树）
2. 点击 **Generate Code** 生成初始化代码（会自动覆盖 `Core/Src` 和 `Core/Inc` 下 CubeMX 管理的文件）
3. 用 VSCode 编辑 `freertos.c` 等业务逻辑文件（在 `/* USER CODE BEGIN */` 和 `/* USER CODE END */` 块内编写）
4. 用 Keil 编译并烧录

> ⚠️ 注意：CubeMX **重新生成代码会覆盖**所有不在 USER CODE 保护块内的内容。如果在 CubeMX 管理的文件中（如 `can.c`、`main.c`）添加了自定义代码，务必放在 USER CODE 块内，或在 Generate Code 后手动重新添加。

### 3.2 ST-Link 使用说明

ST-Link 用于程序烧录和在线调试：
- SWD 接口只需 4 根线：SWCLK、SWDIO、GND、3.3V
- F407 通过 ST-Link 连接电脑（同时供电）
- F103 通过独立 ST-Link 连接电脑（同时供电）
- Keil 中选择对应 ST-Link 驱动即可烧录

### 3.3 CH340 与 USART 串口配置

CH340 USB 转串口模块用于连接 F103 的 USART1 和电脑：
- CH340 TX → F103 PA10（USART1_RX）
- CH340 RX → F103 PA9（USART1_TX）
- GND → 共地
- 波特率：115200，8N1（8 数据位，无校验，1 停止位）
- 串口调试助手推荐：sscom、PuTTY、Xshell 等

### 3.4 Nano DLA 逻辑分析仪 + PulseView 使用说明

逻辑分析仪用于验证 CAN 总线物理层信号：

**硬件连接**：
- DLA GND → 板子 GND（共地）
- CH1 → CAN_H（**只需接一根线**，CAN 为差分信号，单端接 CAN_H 即可解析）

**PulseView 配置**：
1. 采样率：设为 10 MS/s 以上（250kbps 波特率，每位 4μs，10 倍采样足够）
2. 添加 CAN 协议解码器（右键 → 协议 → CAN）
3. 解码器设置：
   - Nominal bitrate：**250000**（根据实际波特率修改）
   - Sample point：**75%**（F103 采样点）或 **83.3%**（F407 采样点）
   - 信号通道：选择 CH1（接 CAN_H 的通道）
4. 点击 Run 开始抓取

> ⚠️ 注意：CAN 解码器的 Sample Point 必须与实际配置匹配，否则 ID 和数据解析全部错误。F103 采样点 = `(1+11)/(1+11+4) = 75%`，F407 采样点 = `(1+9)/(1+9+2) = 83.3%`。

### 4.1 FreeRTOS 任务架构（F407 Master）

| 任务名 | 优先级 | 栈大小 | 周期 | 功能说明 |
|--------|--------|--------|------|---------|
| LedTask | Normal | 512B | 500ms | PC13 LED 翻转，打印心跳 |
| PrintTask | Normal | 512B | 1000ms | 系统状态打印 |
| AdcTask | Normal | 512B | 200ms | 读取 PA1 电位器 ADC 值 |
| I2cTask | Normal | 512B | 2000ms | 读取 AHT20 温湿度 |
| CanTxTask | AboveNormal | 512B | 200ms/2000ms | 发送 CAN 报文（油门/温湿度） |

### 4.2 互斥锁配置

```c
// freertos.c - 全局声明
osMutexId_t printfMutexHandle;           // 保护串口打印，防止多任务输出乱序
osMutexId_t sensorDataMutexHandle;      // 保护共享传感器数据（ADC值、温湿度）

// MX_FREERTOS_Init() - 初始化时创建
printfMutexHandle = osMutexNew(&printfMutex_attributes);
sensorDataMutexHandle = osMutexNew(&sensorDataMutex_attributes);
```

### 4.3 CAN 数据帧格式

```c
// F407 发送 - ID=0x100（油门 ADC）
TxHeader.StdId = 0x100;
TxHeader.DLC = 4;
TxData[2] = (uint8_t)(adc_val >> 8);    // 字节2 = ADC 高8位
TxData[3] = (uint8_t)(adc_val & 0xFF);  // 字节3 = ADC 低8位

// F407 发送 - ID=0x101（温湿度）
TxHeader.StdId = 0x101;
TxHeader.DLC = 8;
TxData[0] = (uint8_t)(temp_int >> 8);   // 字节0 = 温度高8位
TxData[1] = (uint8_t)(temp_int & 0xFF); // 字节1 = 温度低8位
TxData[2] = (uint8_t)(humi_int >> 8);   // 字节2 = 湿度高8位
TxData[3] = (uint8_t)(humi_int & 0xFF); // 字节3 = 湿度低8位
```

### 4.4 F103 CAN 过滤器配置

```c
// main.c - CAN 初始化后配置过滤器
CAN_FilterTypeDef sFilterConfig;
sFilterConfig.FilterBank = 0;
sFilterConfig.FilterMode = CAN_FILTERMODE_IDMASK;
sFilterConfig.FilterScale = CAN_FILTERSCALE_32BIT;
sFilterConfig.FilterIdHigh = 0x0000;
sFilterConfig.FilterIdLow = 0x1000;       // 匹配 0x10xx
sFilterConfig.FilterMaskIdHigh = 0xFFFF;
sFilterConfig.FilterMaskIdLow = 0xFFF0;   // 低4位忽略 → 接收 0x100~0x10F
sFilterConfig.FilterFIFOAssignment = CAN_RX_FIFO0;
sFilterConfig.FilterActivation = ENABLE;
```

### 5.3 2026-04-05｜第三天：CAN 引脚冲突与重映射

**目标**：完成 F103 的 CAN 外设配置，解决 PA11/PA12 被 USB 占用的问题。

**CubeMX 配置过程（F103）**：
1. 打开 `.ioc` → Pinout & Configuration → CAN1 → Parameter Settings
2. 设置波特率：Prescaler = 4，BS1 = 15TQ，BS2 = 2TQ（计划 500kbps）
3. **关键步骤**：Pinout View → CAN1_RX → 选择 **PB8**（自动将 CAN_TX 重映射到 PB9）
4. System Core → GPIO → PB8 模式：Input，PB9 模式：Alternate Push-Pull
5. System Core → NVIC → 开启 **USB_LP_CAN1_RX0_IRQn**（CAN RX0 中断）

**完成内容**：
1. CAN1 重映射到 PB8(RX) / PB9(TX)，避开被 USB 占用的 PA11/PA12
2. 在 `can.c` 的 `HAL_CAN_MspInit()` 中添加重映射代码

**踩过的坑**：

| # | 问题 | 原因 | 解决方法 |
|---|------|------|---------|
| 4 | F103 PA11/PA12 被 USB 电路占用 | 核心板上 USB 接口通过跳线帽连接 PA11/PA12 | 重映射 CAN1 到 PB8/PB9，避开 USB 占用 |
| 5 | PB8 无法配置为 CAN_RX | STM32F103 默认启用完整 JTAG，PB8 是 TDI 引脚，被硬件占用 | 在 CAN 重映射之前调用 `__HAL_AFIO_REMAP_SWJ_NOJTAG()` 禁用 JTAG，仅保留 SWD |

```c
/* can.c - HAL_CAN_MspInit() 中添加 */
__HAL_AFIO_REMAP_SWJ_NOJTAG();  /* 禁用 JTAG，保留 SWD，释放 PB8/PB9 */
__HAL_AFIO_REMAP_CAN1_2();       /* 将 CAN1 重映射到 PB8/PB9 */
```

> 💡 附加说明：CubeMX 中 PB8 设为 Input、PB9 设为 AF_PP 并非错误。F1 系列的 GPIO 配置方式与 F4 不同——CAN_RX 引脚必须设为 Input，CAN_TX 引脚必须设为 AF_PP，配置完全正确。

### 5.4 2026-04-05｜第四天：CAN 波特率时钟配置

**目标**：理解两端 CAN 时钟差异，正确配置波特率参数。

**时钟树分析**：

**F407**：
- HSE = 8MHz → PLL → SYSCLK = 168MHz → APB1 = HCLK/4 = **42MHz**
- CAN1 挂在 APB1 上 → CAN 时钟 = 42MHz
- 原计划：Prescaler = 6，BS1 = 11，BS2 = 2 → 42/6/14 = **500kbps** ✅

**F103**：
- HSE = 8MHz → PLL → SYSCLK = 72MHz → APB1 = HCLK/2 = **36MHz**
- CAN1 挂在 APB1 上 → CAN 时钟 = 36MHz
- 配置：Prescaler = 4，BS1 = 15，BS2 = 2 → 36/4/18 = **500kbps** ✅

**踩过的坑**：

| # | 问题 | 原因 | 解决方法 |
|---|------|------|---------|
| 6 | 误以为 F407 CAN 使用 APB2=84MHz | F4 系列 CAN1 实际挂在 APB1 总线上，不是 APB2 | 确认 CAN1 时钟为 APB1=42MHz，实际波特率 500kbps |

> ⚠️ 注意：虽然两端计算结果都是 500kbps，但 F407 的 APB1 是 42MHz，F103 是 36MHz，实际采样点位置不同，ACK 时隙存在偏差。这个差异在后续调试中才暴露出来，最终通过降低波特率解决。

### 5.5 2026-04-06｜第五天：CAN 通信调试与最终成功

**目标**：调试两端 CAN 通信，解决 Stuff Error 和数据解析问题，实现稳定通信。

**调试过程**：

**第一步**：用逻辑分析仪确认 F407 端发送正常。抓取 CAN_H 信号，解码器设置为 500kbps / 75% 采样点，看到 ID = 0x100 的报文，**但没有 ACK 应答**。

**第二步**：在 F103 端读取 CAN 寄存器诊断状态：
```c
uint32_t esr = hcan.Instance->ESR;  // ESR = 0xFF000023 → LEC = 2 = Stuff Error
```
Stuff Error 说明 F103 检测到了位填充错误。原因是 F407 和 F103 的采样点位置不同，F103 的采样时刻落在了 ACK 时隙，F103 的 TJA1050 驱动 dominant ACK 与 F407 的 recessive ACK 冲突，导致整个帧 CRC 校验失败。

**第三步**：两端同时降低波特率到 250kbps：
- F407：`Prescaler = 14，BS1 = 9，BS2 = 2` → 42/14/12 = 250kbps，采样点 83.3%
- F103：`Prescaler = 9，BS1 = 11，BS2 = 4` → 36/9/16 = 250kbps，采样点 75%

**第四步**：CAN 通信成功后，发现接收到的 ADC 数据全为 0。修改 F103 回调函数，从 rx_data[2] 和 rx_data[3] 读取后，数据完全正确。

**踩过的坑**：

| # | 问题 | 原因 | 解决方法 |
|---|------|------|---------|
| 7 | 500kbps 通信失败（LEC=2 Stuff Error） | F407(42MHz) 和 F103(36MHz) 时钟不同，采样点偏差导致 ACK 时隙冲突 | 两端同时降为 250kbps |
| 8 | 中间波特率 473.7kbps 仍失败 | 不精确的非标准波特率，F407 500kbps 与 F103 473.7kbps 仍有偏差 | 使用精确的 250kbps |
| 9 | ADC 数据全为 0 | F407 发送用字节2-3（`adc_val>>8` 和 `adc_val&0xFF`），F103 接收读字节0-1 | 修改 F103 回调从 `rx_data[2]` 和 `rx_data[3]` 读取 |
| 10 | TJA1050 5V 供电抓不到波形 | 拓展坞供电不稳定，多设备争抢电流 | 改用 3.3V 供电（TJA1050 模块自带稳压电路） |
| 11 | sensorDataMutexHandle 未创建 | `osMutexNew()` 未调用，但变量已声明 | 在 `MX_FREERTOS_Init()` 中添加 `osMutexNew()` |
| 12 | F103 过滤器重复配置 | can.c 和 main.c 各自配置过滤器互相覆盖 | 删除 can.c 中的冗余配置，保留 main.c 中的配置 |

## 七、面试相关知识点清单

### CAN 总线核心知识点
1. **标准帧 vs 扩展帧**：本项目使用标准 11 位 ID（CAN 2.0A），ID 范围 0x000~0x7FF
2. **位填充机制**：连续 5 个相同电平后必须插入一个反向位，否则触发 Stuff Error
3. **ACK 机制**：接收节点在 ACK 时隙拉低总线，发送节点检测到 ACK 才认为传输成功
4. **错误帧**：检测到错误的节点立即发送 6 个 dominant 位打断总线
5. **波特率计算**：`波特率 = CAN时钟 / (Prescaler × (1 + BS1 + BS2))`
6. **采样点位置**：`采样点 = (1 + BS1) / (1 + BS1 + BS2)`，车载标准为 87.5% 左右
7. **过滤器配置**：STM32 硬件过滤器只接收特定 ID 范围，减少 CPU 中断负担
8. **终端电阻**：120Ω 终端电阻吸收总线反射，车载 CAN 总线两端各一颗
9. **差分信号**：CAN_H 和 CAN_L 电压差表示数据，dominant ≈ 2V，recessive = 0V
10. **Bus Off 机制**：发送错误计数超过 255 时进入 Bus Off，需重启恢复

### FreeRTOS 核心知识点
1. **互斥锁创建**：`osMutexNew()` 必须在使用前调用，未创建的互斥锁所有操作均失败
2. **任务优先级**：高于等于 `configMAX_SYSCALL_INTERRUPT_PRIORITY` 的中断不能调用 FreeRTOS API
3. **阻塞延时**：FreeRTOS 任务中必须使用 `osDelay()`，`HAL_Delay()` 会阻塞调度器
4. **堆内存配置**：`configTOTAL_HEAP_SIZE` 决定能创建多少任务和队列

### 嵌入式调试方法
1. **寄存器诊断**：读取外设状态寄存器（ESR/MSR/RF0R）定位 CAN 通信问题
2. **LED 调试**：在中断处理函数中加 `HAL_GPIO_TogglePin()`，通过 LED 闪烁频率判断中断是否触发
3. **逻辑分析仪**：抓取总线信号验证物理层，解析协议验证数据正确性
4. **逐步降速**：通信异常时降低波特率，排除时钟偏差问题
